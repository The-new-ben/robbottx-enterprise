# RobbottX Core 0.1.5 acceptance contract

## Purpose

Version 0.1.5 closes two proven foundation gaps:

1. Canonical fixtures must pass the declared JSON Schema 2020-12 contract, not
   only the custom business-rule validator.
2. Inherited WordPress records must remain recoverable while disappearing from
   public search, feeds, native sitemaps, and indexable output.

The release does not approve the candidate configuration, create commercial
offers, claim physical validation, or delete inherited data.

## Explicit requirements

- Execute the complete canonical JSON Schema against the fixture.
- Retain custom cross-record and publication invariants.
- Reject graph edges whose subject or object cannot be resolved.
- Prove schema and graph failures with mutation tests.
- Exclude the inherited `robot` post type from public discovery.
- Exclude the exact `robots-catalog`, `sample-page`, and `hello-world` records.
- Exclude the exact Uncategorized term and the users provider from native
  sitemaps while preserving future editorial taxonomies.
- Exclude inherited records from unauthenticated REST collections, search, and
  direct `GET` or `HEAD` detail requests.
- Add `noindex,follow` for inherited singular, archive, author, category, and
  exact sample/default routes.
- Preserve WordPress administration, REST administration, cron, WooCommerce,
  account, cart, checkout, media, and future approved RobbottX records.
- Delete no database rows and change no production state during source work.

## Implied dependencies

- JSON Schema format validation must cover `date`, `date-time`, and `uri`.
- Schema errors must include stable instance paths that can be diagnosed.
- Search and feed filtering must target front-end main queries only.
- Sitemap filters must use current WordPress core hook signatures.
- Exact inherited records must be resolved by stable slugs, not numeric IDs.
- Direct REST detail controls must leave writes to WordPress and preserve
  resource-specific editorial capabilities.
- Rank Math robots output must agree with WordPress core robots output.
- The deterministic package allowlist must include every new runtime file.
- Plugin header, version constant, block version, manifest, inventory, and ZIP
  name must agree.

## Proposed additions

- A dedicated schema validator module using the JSON Schema 2020-12 dialect.
- Endpoint reference checks against the union of typed record identifiers.
- A dedicated WordPress public-discovery policy class.
- A PHP harness that simulates search, feed, sitemap, robots, admin, and
  commerce contexts without a production database.

## Unknowns that remain outside this release

- The host currently intercepts `/robots.txt` with an nginx 404. A WordPress
  `robots_txt` filter cannot be accepted as a root-path fix until a live request
  proves that nginx reaches WordPress.
- Rank Math may maintain its own sitemap index. Version 0.1.5 governs native
  WordPress sitemaps and robots output. Live verification must identify every
  active sitemap provider.
- Inherited URLs may already have external links or search history. This release
  uses reversible `noindex,follow`. Redirects or HTTP 410 responses require a
  later URL-by-URL evidence decision.

## Contradictions resolved

- The repository excluded retired assets, but production still exposed retired
  database records. Source exclusion and public discovery are separate gates.
- The contract command reported success, but it had not executed the declared
  JSON Schema. Custom invariants and schema conformance are separate gates.
- Direct production is required, but source review must precede deployment.
  A Git branch is a review boundary, not a staging site.

## Risks and controls

| Risk | Control |
| --- | --- |
| Search loses future catalog records | Remove only the inherited post type and exact default IDs. |
| WooCommerce routes or products are hidden | Guard administration and commerce contexts, and test their preservation. |
| Public REST detail routes bypass collection filters | Gate exact inherited `GET` and `HEAD` routes before dispatch while preserving resource-specific editorial capabilities. |
| Sitemaps expose inherited records through another provider | Remove the post subtype, category taxonomy, and user provider independently. |
| Duplicate or conflicting robots directives | Normalize both WordPress and Rank Math robots arrays. |
| Schema validation accepts malformed dates or URLs | Enable standard format validation and mutation tests. |
| Graph validation rejects legitimate typed endpoints | Resolve the documented union of entity, revision, configuration, BOM, evidence, assertion, rule, and assessment IDs. |
| New runtime file is omitted from the ZIP | Update the deterministic allowlist and inspect the packaged marker. |
| Production cannot be recovered after a bad deploy | Use the temporary admin-gated deploy route, verify independently, and delete the route. |

## Source acceptance evidence

- Full `npm run qa` passes.
- Every changed PHP file passes `php -l`.
- Schema mutation removing a required field fails with a useful path.
- Schema mutation adding an unknown property fails.
- Graph mutation using a nonexistent endpoint fails.
- Search and feed harnesses exclude inherited content while preserving approved
  RobbottX types, products, and media.
- REST harnesses exclude inherited collection, search, and direct detail
  surfaces while preserving arbitrary records, commerce records, writes, and
  resource-specific editorial access.
- Sitemap harness excludes inherited post type, exact default records,
  Uncategorized, and users while preserving product and editorial providers.
- Robots harness returns `noindex,follow` only for the intended public routes.
- Admin and WooCommerce harnesses remain unchanged.
- Public-language and forbidden-input scans pass.
- Two consecutive package builds produce the same ZIP SHA-256.
- ZIP inspection proves the new policy class, version strings, and no secrets.

## Direct-production acceptance evidence

Deployment order is deliberate and verified one version at a time.

1. Confirm the published raw ZIP and inventory hashes.
2. Create one temporary administrator Application Password.
3. Create the temporary, capability-gated deploy route.
4. Install the reviewed ZIP with overwrite enabled.
5. Purge page and object caches.
6. Verify the public health version and rendered-body marker independently.
7. Delete the deploy route and prove it returns 404.
8. Revoke the temporary Application Password.
9. Verify the inherited URL matrix, search, feeds, and every active sitemap.
10. Verify homepage, shop, cart, checkout, account, and admin behavior.
11. Run desktop and mobile visual, accessibility, performance, and console QA.
12. Record exact timestamps, status codes, hashes, screenshots, and limitations.
